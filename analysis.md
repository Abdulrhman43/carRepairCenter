# Car Repair Center - Deep Technical Analysis

Date: 2026-05-25  
Reviewer: Codex  
Scope: Full-stack repository review of backend, frontend, persistence model, security, runtime behavior, maintainability, and verification state.

## Executive Summary

Car Repair Center is a serious full-stack repair-shop management system, not a toy CRUD app. It has a clean three-project .NET backend, a modern React/Vite client, a coherent Arabic RTL user experience, role-based access control, inventory deduction, payment tracking, reporting, and invoice-oriented workflows.

The foundation is good, but the project is not production-ready yet. The highest-risk issues are not visual or structural; they are correctness, security, and verification gaps:

1. The frontend currently does not build because `useRef<ReturnType<typeof setTimeout>>()` is called without an initial value in `RepairOrdersPage.tsx`.
2. Dashboard/report endpoints start several EF Core async queries concurrently on the same scoped `AppDbContext`, which is an invalid EF Core usage pattern and can fail at runtime.
3. JWT signing secrets are committed in `appsettings.json` and duplicated as source-code fallbacks.
4. Sequential business codes are generated by reading the latest row and incrementing in application code, which creates race conditions under concurrent requests.
5. Core business rules are mostly enforced only by UI convention or thin controller checks; negative quantities, invalid discounts, overpayment, and impossible dates can reach the database.
6. There are no automated tests, no API contract generation, no CI, and no reliable safety net for future changes.

My verdict: the codebase has a good MVP/product foundation, but it needs a focused hardening phase before real shop usage with multiple users.


Root cause:

```ts
const discountTimerRef = useRef<ReturnType<typeof setTimeout>>();
```

In React 19 TypeScript typings, `useRef` expects an initial value for this overload. Use:

```ts
const discountTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
```

and guard `clearTimeout` accordingly.

## Backend Architecture Analysis

### What Is Strong

The dependency direction is mostly correct. `Core` has no dependency on EF, ASP.NET, or the API layer. `Infrastructure` references `Core`, and `API` references both. That gives the project a clean starting point for later extraction of services, tests, and use cases.

The domain model covers the main business flow:

```text
Customer -> Vehicle -> RepairOrder
RepairOrder -> RepairOrderService -> Service
RepairOrder -> RepairOrderPart -> InventoryItem
RepairOrder -> Payment
```

The database model includes important uniqueness constraints:

| Entity | Constraint/Index |
|---|---|
| Customer | Unique `CustomerCode`, unique `Phone` |
| Vehicle | Unique `PlateNumber` |
| InventoryItem | Unique `ItemCode` |
| RepairOrder | Unique `OrderCode`, indexes on status/date/customer/vehicle |
| Payment | Indexes on repair order and paid date |

Read-heavy order endpoints use `AsNoTracking()` and `AsSplitQuery()` in the right places to reduce EF tracking overhead and avoid Cartesian explosion when loading multiple collections.

### Architectural Problem

Controllers directly inject `AppDbContext` and contain business workflows:

- Customer code generation.
- Repair order code generation.
- Inventory item code generation.
- Repair order status transitions.
- Inventory deduction/restoration.
- Payment acceptance.
- DTO mapping.
- Validation decisions.

This works for an MVP, but it makes the system hard to test and easy to break. The next architecture step should not be a massive rewrite; it should be targeted extraction of application services around high-risk workflows:

| Service | Owns |
|---|---|
| `RepairOrderWorkflowService` | Create order, change status, add/remove service, add/remove part, discount rules. |
| `InventoryReservationService` | Stock deduction, restoration, stock validation, concurrency handling. |
| `PaymentService` | Payment validation, overpayment rules, payment reversal/voiding later. |
| `BusinessCodeGenerator` | Customer/order/inventory codes with race-safe persistence. |
| `RepairOrderMapper` | Shared DTO projection logic. |

## Critical Findings

### C1 - Frontend Build Is Broken

File: `client/src/features/orders/RepairOrdersPage.tsx:224`

The frontend TypeScript build fails because `useRef` is called without an initial value. This means the client cannot be reliably built for deployment.

Recommended fix:

```ts
const discountTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
```

Then:

```ts
if (discountTimerRef.current) {
  clearTimeout(discountTimerRef.current);
}
```

### C2 - Parallel EF Queries Use The Same DbContext

Files:

- `src/CarRepairCenter.API/Controllers/DashboardController.cs:24-30`
- `src/CarRepairCenter.API/Controllers/DashboardController.cs:88-100`

`DashboardController` starts multiple async EF Core queries and waits for them with `Task.WhenAll`. The intent is performance, but a single `DbContext` instance does not support multiple active operations at the same time.

Risk:

- Runtime `InvalidOperationException`.
- Intermittent dashboard/report failures.
- Hard-to-debug behavior under load.

Recommended fixes:

1. Run these queries sequentially first. This is the safest immediate fix.
2. If parallelism is really needed, use `IDbContextFactory<AppDbContext>` and create a separate context per parallel query.
3. Prefer combining related metrics into fewer SQL projections where possible.

This issue should be fixed before trusting dashboard/report endpoints.

### C3 - JWT Secret Is Committed And Has Code Fallbacks

Files:

- `src/CarRepairCenter.API/appsettings.json`
- `src/CarRepairCenter.API/Program.cs`
- `src/CarRepairCenter.API/Controllers/AuthController.cs`

The same signing key is stored in `appsettings.json` and duplicated as a fallback string in code. If this repository has ever been shared, the token signing secret should be considered compromised.

Recommended fix:

- Remove hardcoded fallback keys.
- Require `Jwt:Key` from environment/user-secrets.
- Use `dotnet user-secrets` for development.
- Use deployment secrets for production.
- Rotate the key immediately for any real deployment.

### C4 - Race Conditions In Business Code Generation

Files:

- `CustomersController.cs`
- `RepairOrdersController.cs`
- `InventoryController.cs`

Pattern:

```csharp
var lastCode = await _db.RepairOrders
    .OrderByDescending(r => r.Id)
    .Select(r => r.OrderCode)
    .FirstOrDefaultAsync();
```

Then the controller increments the number in memory.

Risk:

- Two concurrent requests can generate the same code.
- Unique indexes will throw exceptions.
- Users may see failed creates under normal multi-user use.

Recommended fix options:

1. Add a `BusinessSequence` table and update it inside a transaction.
2. Catch unique constraint exceptions and retry code generation.
3. Use non-sequential readable codes, such as `REP-20260525-8F3A`.
4. Move all code generation into one service so the rule is testable.

### C5 - Inventory Can Go Negative By Design

File: `src/CarRepairCenter.API/Controllers/RepairOrdersController.cs:157-162`

The API warns if requested quantity exceeds stock, but still subtracts it:

```csharp
item.Quantity -= dto.Quantity;
```

This may be intentional for backorders, but the current model does not explicitly represent backordered quantity, reservation status, or inventory adjustment reason. As written, negative stock can be created silently from the API.

Recommended decision:

- If negative stock is allowed, model it explicitly as backorder behavior.
- If not allowed, return `400 BadRequest` when `dto.Quantity > item.Quantity`.
- In both cases, validate `dto.Quantity > 0`.

## High-Priority Issues

### H1 - Validation Is Too Thin

DTOs use `[Required]`, but important numeric and domain rules are missing.

Examples:

| Field/Workflow | Missing Rule |
|---|---|
| `DiscountPercentage` | Must be between 0 and 100. |
| `AddRepairOrderPartDto.Quantity` | Must be greater than 0. |
| `UnitPrice`, `DefaultPrice`, `Payment.Amount` | Must be non-negative or greater than 0 depending on workflow. |
| `Vehicle.Year` | Should be realistic, not future-impossible. |
| `Payment.Amount` | Should not overpay unless overpayment is explicitly supported. |
| `Customer.Phone` | Should be normalized and validated before unique constraint enforcement. |

Recommended fix:

- Add FluentValidation or custom endpoint validation.
- Keep validation rules close to application services, not only UI forms.
- Return consistent `ProblemDetails` responses.

### H2 - No Global Error Handling

There is no global exception middleware. Database constraint exceptions, JSON errors, and unexpected failures will produce inconsistent responses.

Recommended fix:

- Add `UseExceptionHandler`.
- Return RFC 7807 `ProblemDetails`.
- Log exception details server-side only.
- Map common domain exceptions to `400`, `404`, `409`, or `422`.

### H3 - No Pagination On List Endpoints

Endpoints like customers, vehicles, payments, inventory, and repair orders return whole result sets. This is fine for demo data and risky for real shop history.

Recommended contract:

```json
{
  "items": [],
  "page": 1,
  "pageSize": 25,
  "totalCount": 430
}
```

Add pagination and sorting to:

- `GET /api/customers`
- `GET /api/vehicles`
- `GET /api/repairorders`
- `GET /api/payments`
- `GET /api/inventory`
- `GET /api/services`

### H4 - LocalStorage Token Storage Increases XSS Blast Radius

Files:

- `client/src/store/authStore.ts`
- `client/src/services/api.ts`

The frontend stores JWTs in `localStorage`. This is common for small apps, but if an XSS bug is introduced, the token is easy to steal.

Recommended production direction:

- Short-lived access token.
- HttpOnly, Secure, SameSite refresh cookie.
- Refresh endpoint with rotation.
- CSP headers to reduce XSS risk.

### H5 - No Audit Trail Or Soft Delete

The system tracks some creation data, but does not consistently track:

- `UpdatedAt`
- `UpdatedByUserId`
- soft deletion
- payment void/reversal
- inventory adjustment reason
- status transition history

For a repair shop, auditability matters because money, stock, discounts, and customer claims are involved.

Recommended additions:

- Add audit columns to main entities.
- Use EF Core `SaveChangesInterceptor` for common audit fields.
- Add `RepairOrderStatusHistory`.
- Add `InventoryMovement` for stock changes.
- Add payment voiding instead of hard delete.

## Medium-Priority Issues

### M1 - Mapping Logic Is Duplicated

`RepairOrdersController` and `DashboardController` both build `RepairOrderDto` manually. This duplication will drift as the invoice/order shape evolves.

Recommended fix:

- Extract `RepairOrderMapper`.
- Or use projection expressions where possible to keep expensive DTO shaping in SQL.

### M2 - Frontend Uses Browser `alert()`

Several feature pages show errors with `alert()`. That blocks the UI, feels unfinished, and gives no structured feedback.

Recommended fix:

- Add a toast system.
- Standardize API error parsing.
- Surface warning, success, and destructive action messages consistently.

### M3 - Large Frontend Components

Largest frontend files:

| File | Lines |
|---|---:|
| `RepairOrdersPage.tsx` | 760 |
| `CustomersPage.tsx` | 303 |
| `LoginPage.tsx` | 222 |
| `VehiclesPage.tsx` | 222 |

`RepairOrdersPage.tsx` is doing too much: list, filters, create form, details modal, status updates, service addition, part addition, payment registration, discount debounce, print invoice, and data refresh.

Recommended split:

- `RepairOrderList`
- `RepairOrderCreateDialog`
- `RepairOrderDetailsDialog`
- `OrderServicesPanel`
- `OrderPartsPanel`
- `OrderPaymentsPanel`
- `InvoicePrintView`
- `useRepairOrders`

### M4 - Frontend And Backend Contracts Are Manually Mirrored

The TypeScript interfaces in `client/src/types/index.ts` manually mirror C# DTO records in `src/CarRepairCenter.API/DTOs/Dtos.cs`.

Risk:

- Silent drift between backend and frontend.
- Runtime bugs after DTO changes.

Recommended fix:

- Add Swagger/OpenAPI.
- Generate a typed API client.
- Or at least generate TypeScript DTOs from the OpenAPI schema.

### M5 - SQLite Needs Production Configuration

Current connection string:

```json
"DefaultConnection": "Data Source=carrepair.db"
```

SQLite can be excellent for a single-location lightweight app, but the project should explicitly configure it for the expected deployment mode.

Recommended:

- Enable WAL mode.
- Consider busy timeout.
- Consider connection pooling settings.
- Backup the `.db`, `-wal`, and `-shm` files safely.
- Move to PostgreSQL or SQL Server if there will be many concurrent users, remote branches, or heavy reporting.

### M6 - Date Filtering Uses `.Date` In Queries

Patterns like:

```csharp
p.PaidAt.Date == targetDate
```

can be less index-friendly than range filtering.

Preferred:

```csharp
var start = targetDate;
var end = start.AddDays(1);
query.Where(p => p.PaidAt >= start && p.PaidAt < end)
```

This is clearer and more likely to use indexes efficiently.

### M7 - Role Model Is Too Coarse

Current roles are essentially `Admin` and regular authenticated users/receptionists. This may be enough for an MVP, but real shops often need:

- Owner
- Manager
- Receptionist
- Technician
- Accountant
- Inventory manager

Do not add all roles now unless required, but design new authorization policies around permissions rather than scattering role strings.

## Security Review

### Good Security Choices

- ASP.NET Identity is used for password hashing.
- JWT bearer authentication protects API controllers.
- Role-based restrictions exist for admin areas.
- Password length and non-alphanumeric requirement are configured.
- Frontend routes hide admin-only pages.

### Security Gaps

| Risk | Impact | Fix |
|---|---|---|
| Committed JWT key | Token forgery if key leaks | Move to secrets and rotate. |
| Long-lived 7-day access token | Stolen token remains useful | Use short access token + refresh flow. |
| Token in localStorage | XSS can steal token | Use HttpOnly refresh cookie and CSP. |
| No rate limiting | Login and API abuse | Add ASP.NET rate limiting. |
| No lockout on login call | Brute force risk | Pass `lockoutOnFailure: true` and configure lockout. |
| No HTTPS/HSTS production branch | Transport risk | Add HTTPS redirect/HSTS outside development. |
| Broad `AllowedHosts` | Deployment hardening gap | Set real hosts in production. |

## Domain Model Review

### Strong Domain Coverage

The project captures the core shop lifecycle:

```text
Register customer
Add vehicle
Open repair order
Attach services
Consume parts
Track status
Collect payments
Print invoice
Report revenue and outstanding balance
```

This is the right product spine.

### Domain Rules To Clarify

The code should explicitly answer these business questions:

1. Can stock go below zero?
2. Can a payment exceed remaining balance?
3. Can a delivered order be edited?
4. Can a discount be changed after payment?
5. Can a payment be voided, and who can do it?
6. Can a customer phone number be reused after deletion?
7. Is `PlateNumber` globally unique forever?
8. Should order totals be recalculated live or snapshotted at invoice time?
9. Should part price use current inventory price or historical price at time of addition?
10. Are statuses strictly forward-only, or can an admin roll them back?

The current implementation allows too many of these to be accidental rather than deliberate.

## Frontend Analysis

### Strengths

- Feature-based page organization is easy to navigate.
- Zustand is a good lightweight fit for auth/theme state.
- Axios interceptor centralizes JWT attachment and 401 redirects.
- Arabic RTL appears to be first-class, not an afterthought.
- The UI has useful operational screens: dashboard, customers, vehicles, orders, inventory, payments, reports.
- Print invoice support is a real product feature.

### Weaknesses

- The client build currently fails.
- Error handling is mostly `alert()`.
- API calls are embedded directly in page components.
- Large pages mix data fetching, mutation logic, modal state, and rendering.
- Loading and disabled states are inconsistent.
- Token expiration is trusted through local hydrated state; the app does not appear to decode/expire-check JWT on hydration.
- All Arabic copy is hardcoded, so adding English later would be expensive.

Recommended next frontend architecture:

```text
features/orders/
  api.ts
  hooks.ts
  components/
  RepairOrdersPage.tsx
```

Start with the orders feature because it has the most business complexity.

## Final Engineering Verdict

This project is ahead of a typical MVP in product shape and UX completeness. The repair-order workflow, inventory connection, payment records, Arabic RTL interface, and reporting pages show real product thinking.

The main issue is that the codebase has grown past the point where controller-driven business logic and manual verification are enough. The next step is not a redesign; it is hardening. Fix the build, remove secrets, correct EF concurrency, add validation, and put tests around the money/stock/order workflows. After that, this becomes a much safer base to extend.
